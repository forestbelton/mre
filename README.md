# Monster Rancher Explorer — disassembly

A work-in-progress disassembly of **Monster Rancher Explorer** (テクモ / Tecmo,
Game Boy Color). The build reassembles the source tree back into a ROM that is
**byte-for-byte identical** to the original (verified by sha256).

This is a **source project**, not just a region map. The aim is to recover
something close to the inputs the original developers built from: assets in
editable forms (PNGs, Tiled `.tmx`, structured data) with the tooling to compile
them back to bytes, and code/data that references everything by symbolic label
rather than raw address. See [`docs/philosophy.md`](docs/philosophy.md) for the
guiding principles — in particular the litmus test "could we drop the section
offsets and still build?" while byte-exactness remains the actual goal.

## Status

`make verify` reproduces the original ROM exactly:

```
sha256  8f66b5972bf76ed15985815ccdecc459fab9e84221454139b05d1d6654b69e7a
```

Most code and data have been carved out of one monolithic `analyzed.asm` into
named files, with large portions relabeled and many graphics migrated to
editable PNG/`.tmx` assets. Plenty remains as raw `db`/`INCBIN` data pending
further understanding — see the `*_todo.md` notes under [`docs/`](docs).

No ROM is committed, but nothing here needs one: the source builds a complete,
byte-identical ROM on its own, and `make verify` confirms the built ROM matches
the known hash.

## Requirements

- [RGBDS](https://rgbds.gbdev.io/) (`rgbasm`/`rgblink`/`rgbfix`/`rgbgfx`) —
  developed against **v1.0.1**.
- **Python 3** with **Pillow** and **PyYAML** (for the asset/build tooling under
  `tools/`).
- `make`.

## Building

```sh
make           # build the ROM and verify its sha256 (default target)
make rom       # just assemble + link -> build/rom.gbc
make clean     # remove build artifacts (does not touch src/)
```

The source tree under `src/` is the hand-maintained truth; the build only
assembles it. Editable graphics assets (`assets/`, laid out to mirror `src/gfx`)
are compiled to their ROM bytes by `tools/pngasset.py` (driven by
`assets/assets.yaml`), and floor records are generated from JSON by
`tools/build_room.py` — both at build time. The result lands in `build/`
(git-ignored).

## Repository layout

| Path | What |
|---|---|
| `src/` | Hand-maintained assembly: code + data carved per bank/subsystem, plus `layout/` floor JSON. |
| `assets/` | Editable graphics sources (indexed PNGs, Tiled `.tmx`) mirroring `src/gfx`. |
| `include/` | Shared `.inc` macros/constants (`hardware.inc`, `enum.inc`, `monster.inc`, …). |
| `tools/` | Build + reverse-engineering tooling (asset compilers, carving helpers, song export, …). |
| `docs/` | Subsystem notes, the bank map, and the project philosophy. |
| `layout.link` | Linker script; sections migrate off their `ROMX[$addr]` offsets into here as they're understood. |
| `Makefile` | Build orchestration (`verify` / `rom` / `clean`). |
| `build/` | Generated artifacts: object files, compiled assets, `rom.gbc`, the sym/map. |

## The asset pipeline

Each PNG-driven asset is one entry in [`assets/assets.yaml`](assets/assets.yaml).
`tools/buildassets.py` reads that file and runs `tools/pngasset.py` per asset,
emitting the ROM components under `build/assets/<asset>/`, which the assembly
`INCBIN`s. Everything round-trips byte-exact. Asset *modes* include full-screen
colour `screen`s, grayscale `portrait`s, summon `scene`s, and flat OBJ
`sprite` sheets; see [`docs/gfx_assets.md`](docs/gfx_assets.md) and
[`docs/asset_source_model.md`](docs/asset_source_model.md).
